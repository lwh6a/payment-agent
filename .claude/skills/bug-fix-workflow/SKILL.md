# Bug 修复闭环

从工单到代码修复、发版验证、日志确认、回写工单的完整闭环，一条指令触发全流程，不在各工具后台之间手动切换。

**原则**：关闭 Bug 的动作永远属于测试——AI 只把状态推到"已修复"，测试验证通过后由测试关闭。

**前置条件**：项目根目录有 `devops-config.json`（按需要含 jenkins / cls / mysql / xxljob / rocketmq 段，模板见 `devops-config.example.json`）；用云效管理工单时需 `.mcp.json` 的云效 MCP（环境变量 `YUNXIAO_ACCESS_TOKEN`）。

## 流程

### Step 1：读取 Bug 详情

- 有云效：通过云效 MCP 按编号读标题、描述、复现步骤、当前状态。
- 无工单系统：由用户直接描述现象和复现步骤。
- 输出 Bug 摘要给用户确认后再动手。

### Step 2：定位代码 + 按需取证

根据描述中的关键词（接口路径、类名、错误信息）搜索代码；需要环境现场时按类型组合调用取证 skill：

| Bug 类型 | 取证 skill |
|---|---|
| 报错类 | cls-logs（查异常栈和上下文日志） |
| 数据类 | mysql-query（只读查数据现场） |
| 消息类 | mq-inspect（消息是否发出/被消费/堆积） |
| 定时类 | xxljob-ops（任务执行记录和调度结果） |

输出：根因 + 证据（日志/数据/消息轨迹）+ 涉及文件 + 修复方案。**等用户确认后再改代码**。

### Step 3：修复代码

最小必要改动，不改无关代码；输出 diff 给用户确认。

### Step 3.5：提交与推送

- 提交到 `bugfix/<编号>-<简述>` 分支，禁止直接 push 主干/部署分支。
- 创建 MR/PR 走评审；评审不省略。

### Step 4：发版

1. 先按 `deploy-checklist` 生成 SIT 检查单，确认环境就绪。
2. 调用 `jenkins-deploy` trigger（**写操作，须用户确认**），脚本自动轮询到构建结束。
3. 发版失败 → 输出失败原因，停止流程，不做后续验证和回写。

### Step 5：日志验证

等 30 秒（Pod 滚动），用 `cls-logs` 查最近 15 分钟 ERROR：

- 无新增 ERROR → ✅ 验证通过。
- 有 ERROR → 输出详情，标注"需人工确认"，不直接判定失败。

### Step 6：回写工单

验证通过后把工单状态置为"已修复"，附修复说明和发版记录；等测试验证后由测试关闭。

## 日常速用

| 你想做什么 | 对 AI 说的话 |
|---|---|
| 完整修复 | "修复 BUG-123" |
| 只看不修 | "查看 BUG-123" |
| 只发版 | "发版 SIT" |
| 只查日志 | "查 SIT 日志 error" |
| 发版后验证 | "验证发版结果" |
| 中途接入 | "BUG-123 我改好了，帮我发版" |
